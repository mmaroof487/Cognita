## 🤖 Agent Architecture Implemented

### Nodes (4 total)

#### 1. **Collector Node** (`app/agents/nodes/collector.py`)

- **Purpose:** Read GitHub data from database; normalize commits, PRs, developers
- **Input:** tenant_id, org_id, window_start, window_end
- **Output:** commits[], prs[], developers[] (normalized dictionaries)
- **Logic:** Pure deterministic DB queries with SQLAlchemy
- **Key Features:**
  - Fetches commits within time window from TimescaleDB
  - Normalizes PR data (merges, review comments, merge time)
  - Deduplicates developers
  - Handles missing author data gracefully

#### 2. **Analyst Node** (`app/agents/nodes/analyst.py`)

- **Purpose:** Compute metrics and detect anomalies
- **Input:** commits[], prs[], developers[]
- **Output:** developer_metrics{}, team_metrics{}, anomalies[]
- **Logic:** Pure Python (no LLM, no DB)
- **Metrics Computed:**
  - Per-developer: commit_count, additions/deletions, PR stats
  - Team-level: totals, averages, churn ratio
- **Anomalies Detected:**
  1. **burnout_risk:** >15 commits/week + slow PRs (>4h avg merge time)
  2. **high_churn:** Deletions > 2× additions (suggests large refactors)
  3. **slow_review:** PRs taking >72h to merge

#### 3. **InsightAgent Node** (`app/agents/nodes/insight.py`)

- **Purpose:** Generate human-readable insights using Claude LLM
- **Input:** anomalies[], team_metrics{}, developer_metrics{}
- **Output:** insights[] (generated by Claude)
- **Claude Integration:**
  - Model: claude-3-5-sonnet-20241022
  - Prompt: Contextual analysis of anomalies + metrics
  - Output: JSON with title, explanation, recommended_action, severity
- **Error Handling:** Graceful fallback if Claude API fails

#### 4. **ActionAgent Node** (`app/agents/nodes/action.py`)

- **Purpose:** Create actionable items and apply HITL gates
- **Input:** insights[], anomalies[]
- **Output:** actions_queued[] (with status=pending for HITL approval)
- **Actions Generated:**
  - **create_jira:** Jira tickets from templates (HITL gate)
  - **insight_created:** DB write of insights (no gate needed)
- **DB Writes:**
  - Insight records created (insight_type, severity, score, metadata)
  - AgentAction records created with status=pending
  - AuditLog entries written (immutable append-only)

### StateGraph (`app/agents/graph.py`)

```
START
  ↓
Collector (DB read, normalize events)
  ↓
Analyst (Pure Python metrics, anomaly detection)
  ↓
InsightAgent (Claude LLM, generate insights)
  ↓
ActionAgent (Create Jira tickets, DB writes, HITL gate)
  ↓
END
```

**Features:**

- Checkpointing support (MemorySaver for dev, AsyncPostgresSaver ready for prod)
- Error handling and retries via Celery
- State accumulation (insights[], actions_queued[] use operator.add reducer)
- Thread-safe execution

## 🔄 Celery Task Integration

### Task 1: `run_weekly_analysis` (Scheduled)

- **Trigger:** Every Monday 9 AM UTC (beat schedule)
- **Logic:**
  1. Fetch all tenants from DB
  2. For each tenant, fetch all orgs
  3. For each org, queue `run_org_analysis` task
- **Queue:** Redis
- **Status Tracking:** Logged to stdout

### Task 2: `run_org_analysis` (On-demand + Scheduled)

- **Trigger:** Via `run_org_analysis.delay(tenant_id, org_id)` or beat schedule
- **Logic:**
  1. Create AgentRun record (thread_id, status=running)
  2. Build LangGraph with session + Anthropic client
  3. Invoke graph with 7-day window
  4. Update AgentRun record with results (status=completed, tokens, cost)
  5. On error: Update with status=failed, error message, retry with exponential backoff
- **Retries:** 3 retries with 2^retry_count second delay (2s, 4s, 8s)
- **Timeouts:** Soft 25min, hard 30min

**Async/Sync Bridge:**

- Celery tasks are sync (blocking) but invoke async code via `asyncio.run_until_complete()`
- Proper event loop handling for clean shutdown

## 🌐 API Endpoints (New in Week 2)

### 1. `POST /api/v1/orgs/{org_id}/agent-runs`

**Trigger Manual Analysis**

- Request: `{ "analysis_window_days": 7 }`
- Response: `{ "status": "queued", "task_id": "...", "org_id": "..." }`
- Logic: Validates org ownership, queues Celery task

### 2. `GET /api/v1/orgs/{org_id}/agent-runs`

**List Agent Runs (Paginated)**

- Query Params: `limit` (1-100, default 10), `offset` (default 0)
- Response: List of AgentRunResponse objects (id, status, window, counts)
- Pagination: Supported via limit/offset

### 3. `GET /api/v1/agent-runs/{run_id}`

**Get Run Details with Insights & Actions**

- Response:
  ```json
  {
    "run": {
      "id": "...",
      "status": "completed|running|failed",
      "insights_count": 3,
      "actions_count": 5,
      ...
    },
    "insights": [{"id": "...", "title": "...", "severity": "high", ...}],
    "actions": [{"id": "...", "action_type": "create_jira", "status": "pending", ...}]
  }
  ```

**Dependency Injection:**

- `get_current_tenant`: Extracts tenant from JWT (user context)
- `get_async_session`: Provides DB session
- Multi-tenancy: All queries filtered by tenant_id (app-layer RLS)

## 🔐 Security & Multi-Tenancy

**Tenant Isolation:**

- All queries filtered by `tenant_id` at ORM layer
- AgentRun linked to tenant (FK constraint)
- Insights/Actions scoped to tenant
- API endpoints validate `get_current_tenant` before returning data

**HITL Gate (Human-in-the-Loop):**

- AgentAction records created with `status="pending"`
- Awaits manual approval via future `/actions/{id}/approve` endpoint (Week 3)
- Audit log tracks all decisions (actor, action, entity, diff)

**Encryption:**

- Jira template payloads stored in JSONB
- Credentials (API tokens) encrypted at app layer (Fernet)
- Anthropic API key from env var (never logged)
